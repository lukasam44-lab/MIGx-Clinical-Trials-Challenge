# MIGx Clinical Trials — Data Quality & Analytics Challenge

A data quality assessment and analysis of clinical trial data ingested live from the
ClinicalTrials.gov v2 API, profiled and validated against explicit quality rules, then
analyzed to answer five business questions — with data quality caveats woven throughout.

## Approach

Data was ingested from the ClinicalTrials.gov v2 API (Option 4), transformed into a 7-table
normalized SQLite schema, profiled across six quality dimensions, validated by a 12-test
automated suite, and analyzed with SQL and Python. Guiding principle: **assess data you
don't fully trust before drawing conclusions from it.**

## Key Findings

1. **Completion rates are confounded by unverifiable status.** 16.8% of trials have
   status = UNKNOWN; depending on how these are treated, the highest-completing phase flips
   from Phase 1 to Phase 4. Completion metrics must be reported with the unknown proportion
   disclosed.
2. **Mixed date formats limit duration analysis.** 41.8% of studies carry month-precision
   dates, reducing usable duration data by ~45%.
3. **Data is largely clean because it's API-sourced** — controlled-vocabulary fields showed
   no inconsistency, unlike free-text condition names (COVID-19 fragmented across 3 labels).
4. **Enrollment outliers are legitimate, not errors** — large observational registries and
   vaccine trials, distinguished from implausible values via stratified IQR detection.

## Setup & Execution

Requires Python 3.12. SQLite is built into Python — no database server needed.

​```bash
# 1. Environment
python -m venv .venv
.venv\Scripts\activate            # Windows
# source .venv/bin/activate       # macOS/Linux
pip install -r requirements.txt

# 2. Build the database (run in order; idempotent — safe to rerun)
python scripts/Create_database.py     # initialize schema
python scripts/fetch_studies.py       # pull 500 studies from the live API
python scripts/transform_load.py      # transform + load into SQLite

# 3. Run the data quality test suite
python scripts/data_quality_tests.py  # writes docs/defect_log.md

# 4. Explore the analysis — open notebooks in order: 01 → 02 → 03
​```

Note: `data/` is git-ignored (raw JSON and the database are regenerated by the scripts above).
The repository ships the code that produces the data, not the data itself.

## Technical Decisions & Trade-offs

- **SQLite over PostgreSQL:** zero-config reproducibility; reviewer needs only Python.
  Production would use PostgreSQL for concurrency and scale.
- **FK enforcement disabled during load:** deliberate — surfaces quality issues (orphans) in
  testing rather than silently rejecting them at insert (the "investigate before cleaning"
  principle).
- **Raw data git-ignored:** the pipeline regenerates it; repo ships the recipe, not the data.
- **IQR over z-score for outliers:** enrollment is right-skewed / log-normal (mean 30× median);
  z-scores are corrupted by the tail.
- **Faithful ingestion:** source imperfections (partial dates) preserved, not silently fixed.

## Bonus Questions

### 1. Data Quality at Scale

If this pipeline ran daily with new incoming data, I would run the existing test suite
(`data_quality_tests.py`) as a scheduled job on each refresh, extended with: (a) volume
checks — row-count deltas against an expected range to catch partial or failed loads;
(b) distribution drift — alerts if a categorical field's value mix shifts beyond a
threshold, catching upstream schema or source changes; (c) freshness checks — confirming
the most recent records fall within an expected recency window; and (d) the referential-
integrity, format, and cross-field tests already built. Results would be written to a
time-series log rather than a pass/fail snapshot, so trends (e.g. a rising partial-date
rate) become visible before they become problems.

### 2. Test Automation

I would structure the suite as a gate in the pipeline: ETL → tests → conditional promotion.
Tooling: the existing Python suite wrapped in a runner (pytest, or an orchestrator such as
Airflow or Prefect for scheduling and dependency management). Coverage: every quality rule
maps to a test case, as already built. Thresholds would be tiered — Critical failures
(orphaned keys, duplicate primary keys) block promotion of the data; High failures alert and
require sign-off; Low/Informational failures log only. Alerting: Critical and High failures
notify via Slack or email with the defect log attached. The suite is already rerunnable and
side-effect-free (read-only), which is the precondition for safe automation.

### 3. Compliance Considerations (GxP)

In a GxP-regulated environment I would add: (a) a full audit trail — every transformation
logged with timestamp, source, and operator (the `created_at`/`updated_at` columns are a
starting point); (b) validation documentation — IQ/OQ/PQ evidence that the pipeline performs
as specified; (c) formal change control on all code and schema, versioned in Git with
documented change records; (d) end-to-end data lineage — traceability from each reported
figure back to its source records; (e) reproducibility — a deterministic, re-runnable
pipeline (the idempotent design already supports this); (f) access controls and electronic
signatures on any data modification; and (g) a strict no-silent-mutation policy — every
cleaning or transformation decision documented, as practiced in this project's transformation
log.

### 4. Stakeholder Communication

I would adapt the same findings to two altitudes. For a **non-technical executive**: lead with
business risk and a single headline — e.g. "16.8% of trials have unverifiable status, so
completion-rate reporting is unreliable without disclosure." One or two charts, no methodology,
a clear recommendation with its cost and benefit. For a **clinical operations manager**: more
operational detail — which trials, sites, and sponsors have stale or incomplete records; the
specific defects and their remediation steps; the drill-down data they would act on. Same
findings, different depth: the executive needs the decision, the operations manager needs the
worklist.

### 5. Self-Service Analytics

To let stakeholders explore the data safely without my involvement, I would provide: (a) a
cleaned, documented analytical layer (SQL views or a semantic model) with quality flags baked
in — for example an `is_full_date_precision` flag and an `is_analyzable_phase` flag — so users
cannot unknowingly misuse imprecise data; (b) a data dictionary plus the profiling and defect
documentation, so limitations are transparent; (c) a lightweight BI layer (Power BI or Looker)
with pre-built, quality-aware measures; and (d) guardrails — default filters that exclude
UNKNOWN-status or partial-date records unless the user deliberately includes them. The design
principle is to make the safe path the default path.

### 6. Advanced Analytics

Several models could add value: (a) **trial completion prediction** — classifying the
likelihood a trial completes from phase, enrollment, sponsor type, and design (structurally
similar to a loan-default classification); (b) **enrollment forecasting** — predicting final
enrollment from early recruitment trajectory and trial characteristics; (c) **duration
estimation** — regression on phase, therapeutic area, and design to predict trial length
(which would first require resolving the date-precision defect); and (d) **anomaly detection** —
unsupervised flagging of unusual records for QA triage, extending the rules-based tests into
learned patterns. Critically, all of these require the data quality issues to be resolved
first — a model trained on confounded data (for example, UNKNOWN-status trials) would learn
the artifacts rather than the signal.
